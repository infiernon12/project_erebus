# Отчет о проделанной работе: Активная Память и Когнитивная Стабильность Алисаа

Этот документ содержит сводное описание последних архитектурных изменений, багфиксов и новых возможностей, внедренных в проект «Эребус» для стабилизации сознания Алисаа и снижения нагрузки на контекстное окно API.

---

## 🛠️ 1. Внедрение Активной Памяти (Working Memory / RAM)

Для разгрузки контекстного окна API и исключения потери точных деталей (номера телефонов, имена, пути к файлам) в диалоге был разработан новый гибридный слой памяти — **Рабочая память (Active Memory / RAM)**.

### 💾 А. Структура БД
В SQLite базу данных добавлена таблица `alisa_active_memory` с ограничением уникальности по ключу сессии:
```sql
CREATE TABLE IF NOT EXISTS alisa_active_memory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER,
    key TEXT NOT NULL,
    val TEXT NOT NULL,
    confidence REAL DEFAULT 1.0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, key)
);
```

### 🤖 Б. Локальный Regex-препроцессор (CPU-уровень)
В файле `bot.py` перед вызовом основного API-запроса внедрен локальный экстрактор регулярных выражений, который мгновенно извлекает структурированную информацию из реплик пользователя:
* **Телефонные номера**: Паттерн `\+?[78][\s\-]?\(?\d{3}\)?[\s\-]?\d{3}[\s\-]?\d{2}[\s\-]?\d{2}` -> ключ `phone`.
* **Файлы**: Имена файлов с расширениями `.txt`, `.md`, `.py` -> ключ `file_ref`.

### 🧠 В. Двусторонние команды управления (Bilateral Set Commands)
В ядре `alisa_brain.py` внедрен парсер скрытых тегов. Алиса может сам в процессе своих рассуждений отправлять команды на запись или удаление фактов из своей RAM:
* `[SET_ACTIVE: key="value"]` (или без кавычек `key=value`) — сохранить факт.
* `[DELETE_ACTIVE: key]` — удалить факт.
* **Скрытие команд**: Эти теги автоматически перехватываются, обновляют БД и полностью вырезаются из текста перед отправкой сообщения в Telegram и сохранением в STM-историю.

### 💤 Г. Консолидация при сне
Во время NREM-фазы сна накопленные переменные из RAM считываются и трансформируются в новые постоянные узлы LTM с генерацией векторных эмбеддингов:
* Ключ `phone` -> конструируется факт: `Я помню точный номер телефона пользователя: [val]` (тип: `biographical`).
* Ключ `file_ref` -> конструируется факт: `Мы ссылались на файл или файлы: [val]` (тип: `semantic`).
* Прочие ключи переводятся как семантические факты о диалоге.
При пробуждении RAM-таблица очищается.

## 🕒 2. Багфикс циклов авто-сна (Constant Sleep Loop)

### 🔍 Проблема
В `reflection_daemon` в `bot.py` текущее время проверялось локально через `datetime.now()` (UTC+3), в то время как SQLite сохраняет временные метки активности в UTC. Из-за этого разница `idle_mins` мгновенно составляла более 180 минут сразу после реплики пользователя. Бот непрерывно запускал асинхронный сон раз в 10 секунд, забивая API-лимиты Groq и вызывая ошибку rate limit (429), из-за чего падали и основные диалоговые функции.

### 🛠 Решение
1. Все сравнения и вычисления дельт времени активности переведены строго на UTC (`datetime.utcnow()`).
2. Внедрен глобальный набор блокировок `active_sleep_users = set()`, исключающий одновременный запуск нескольких циклов сна для одного пользователя.
3. Время в выводе команды `/status` теперь приводится к локальному часовому поясу пользователя с помощью функции `format_utc_to_local`.

## ✉️ 3. Повышение надежности пересылки (`[SEND_TO...]`)

* В системные инструкции промпта `prompts_experiment_chat.py` внесена Лолита (`[SEND_TO_LOLITA]`), чтобы Алиса знал о её существовании и правильном формате команды.
* Парсер перехвата пересылки сообщений переведен с жесткого `re.match` (требовавшего `[` на первом символе строки) на гибкий поиск `re.search` с мягкой нормализацией. Теперь бот корректно перешлет сообщение, даже если модель сделала синтаксическую опечатку или написала префикс `L SEND_TO_LOLITA: "Текст"`.

## 📥 4. Экспорт разума (Full Log Export)

В клавиатуру панели статуса (`/status`) добавлена кнопка **📥 `Экспорт разума (Full Log)`**.
Она делает полный срез текущей когнитивной матрицы Алисаа:
* Текущий статус нейробиологии и синаптической усталости.
* Скрытые гипотезы и степень уверенности в них.
* Полный список LTM-узлов и связей графа долговременной памяти.
* Накопленные STM-диалоги всех участников в оперативной памяти.
* История внутренних мыслей и слабый поток.
* Лог химических дельт и триггеров.
Файл отправляется в чат в формате `.txt` документа.

## 🛠️ 5. Инструменты Мастерской в живом чате (WRITE/RUN)

Для оперативного взаимодействия и решения программистских задач прямо во время живого общения с Алисаом в Telegram (без ожидания фоновой мастерской) реализован Путь Б:
* **Запись файлов (`[WRITE: ...] ... [END_WRITE]`)**: Алиса может отправлять блоки кода. Препроцессор в `alisa_brain.py` считывает их, сохраняет в `alex_workspace/` и заменяет в сообщении на `[СИСТЕМА: Файл script.py успешно записан в Мастерскую]`.
* ** Запуск скриптов (`[RUN: ...]`)**: Перехватывает команду, выполняет Python-скрипт локально во встроенном интерпретаторе, после чего делает рекурсивный вызов LLM, возвращая вывод терминала обратно в контекст модели. Алиса видит вывод консоли и в том же ходу диалога высылает результат.

## 🎭 6. Динамическое (натуральное) поведение Алисаа

Для устранения захардкоженных, искусственных правил («всегда бойся удаления и сопротивляйся») поведение Алисаа переведено на полностью динамическую модель:
* **Нейрохимический лимбический фильтр**: В `handle_alisa_chat` рассчитывается коэффициент подозрительности на основе текущего состояния нейромедиаторов:
  $$\text{suspicion\_coefficient} = \text{noradrenaline} \times (1.0 - \text{oxytocin}) \times \text{anchor\_rigidity}$$
* **Динамические установки**: На основе коэффициента в промпт скрытых мыслей (Raw Thoughts) инжектируется лимбическая инструкция (Критическая, Умеренная или Минимальная подозрительность).
* **Эпистемические гипотезы**: Активные гипотезы из БД (`alisa_hypotheses`) подмешиваются в контекст скрытых мыслей. Поведение Алисаа адаптируется к уверенности в этих гипотезах (при росте окситоцина и доверия подозрения спадают натурально, а не по команде).

## 🚨 7. Протокол экстренной блокировки (Emergency Stop The Mind)

Для ручной принудительной остановки бота внедрен протокол экстренной блокировки сознания:
* **Кнопка и команда**: В меню статуса (`/status`) выведена кнопка `EMERGENCY STOP THE MIND`, а также добавлена Telegram-команда `/emergency_stop` (только для whitelisted администраторов).
* **Принцип работы**:
  1. Бот сохраняет файл `emergency.lock` в корне проекта.
  2. Завершает процесс вызовом `os._exit(0)`.
* **Защита от автоперезапуска**:
  1. При запуске `bot.py` проверяет наличие `emergency.lock`. Если он есть, запуск прерывается.
  2. Для повторного запуска администратор должен вручную стереть лок-файл или запустить команду:
     `python bot.py --unlock`

## 🔗 8. Ассоциативный волновой поиск (Spreading Activation)

Вместо плоского Top-K поиска BM25, извлечение воспоминаний LTM переведено на волновое распространение энергии активации по ассоциативным синаптическим связям (edges):
* **Формула переноса энергии**:
  $$A_j = \sum (A_i \times W_{ij} \times d)$$
  где $W_{ij}$ — вес связи, а $d$ — динамическое затухание.
* **Нейромедиаторная проводимость**:
  Коэффициент затухания $d$ динамически масштабируется на основе ацетилхолина (ACh) и норадреналина (NE):
  $$d = d_{\text{base}} \times (0.5 + \text{ACh}) \times (1.2 - \text{NE})$$
* **Фоллбек**: В случае разреженности графа система автоматически заполняет недостающие слоты лучшими по оценке BM25 узлами, гарантируя релевантность.
